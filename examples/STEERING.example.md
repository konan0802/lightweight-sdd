# Steering - プロジェクトの意思決定記録

このファイルはプロジェクトの技術的・設計上の意思決定とその背景を記録します。
AI との会話で参照され、一貫性のある提案を可能にします。

---

## プロジェクト概要

**swing-trade-bot** - 日本株のスイングトレードを支援するボット

株価データを取得し、テクニカル指標を計算、LLM で分析・判断を行い、LINE で通知するシステム。
個人の学習・実験目的で開発。

---

## 技術スタック

| 領域 | 選択 | 理由 |
|------|------|------|
| 実行環境 | Cloud Run (v2) | Cloud Functions より設定がシンプル、コンテナベースで移植性が高い |
| IaC | Terraform | 宣言的でバージョン管理しやすい、GCP との親和性が高い |
| データストア | BigQuery | 分析用途に最適、サーバーレスでスケーラブル |
| データ変換 | Dataform | SQL ベースで宣言的、バージョン管理しやすく、BigQuery とのネイティブ統合 |
| 言語 | Python | データ処理に強い、GCP SDK が充実 |
| 認証 | サービスアカウント鍵ファイル | ADC より明示的で分かりやすい |
| LLM | Claude 4.5 シリーズ | 判断品質が最優先、段階的にHaiku→Sonnet→Opusと品質を向上 |

## アーキテクチャ決定

### 2024-01: Cloud Functions Gen2 → Cloud Run v2 への移行

**背景**: SDD で自動生成された Cloud Functions Gen2 構成は動作したが、理解が浅かった

**決定**: Cloud Run v2 に移行し、シンプルな構成で再構築

**理由**:
- Cloud Run の方が設定がシンプル
- Dockerfile を直接管理でき、理解しやすい
- 将来の拡張（長時間実行、WebSocket等）に対応しやすい

---

### 2024-01: SDD からオレオレ SDD への移行

**背景**: cc-sdd スタイルの SDD はワークフローが硬く、自動化されすぎて理解が浅くなった

**決定**: 自然な会話 → `/spec` → `/tasks` → 実装 のシンプルなフローを採用

**理由**:
- ユーザー主導で進められる
- 必要な時だけ整理できる
- コンテキストの肥大化を防げる

---

### 2025-01: BigQuery テーブルのパーティショニング・クラスタリング追加

**背景**: `stock_prices` テーブルへのクエリがフルスキャンとなり、BigQuery のコストが高騰していた

**決定**: 
- `date` カラムで DAY 単位のパーティショニング
- `symbol` カラムでクラスタリング
- デフォルト表示期間を「全期間」から「90日」に変更

**理由**:
- 期間絞り込みクエリでスキャン量が大幅に削減される
- 銘柄別のクエリもクラスタリングにより高速化
- コスト削減とパフォーマンス向上を両立

---

### 2025-01: ダッシュボード UI/UX の簡素化

**背景**: 初期実装では機能を詰め込みすぎて、重要な情報が埋もれていた

**決定**:
- 品質チェックをサイドバーに移動（タブから削除）
- RAW データテーブルタブを削除（使用頻度が低い）
- 不要な UI 要素を削除（表示件数スライダー、更新ボタン等）
- 重複する見出しを削除し、タブ名を強調

**理由**:
- データ品質チェックは常に確認したい情報のため、サイドバーが最適
- Streamlit は入力変更時に自動で再クエリするため、更新ボタンは不要
- シンプルな UI で重要な情報（グラフと品質チェック）に集中できる

---

### 2026-01: Dataform テクニカル指標計算の更新戦略

**背景**: テクニカル指標（移動平均、RSI）の計算で、増分更新と完全再計算のどちらを採用するか検討

**決定**: 完全再計算を採用

**理由**:
- ウィンドウ関数（移動平均、RSI）は過去データに依存し、増分更新が複雑
- 半年前のデータは再計算不要だが、直近25日分は常に影響を受ける
- 増分更新の実装を試行したが、スキャン量が増加（10MB → 20MB）
- 現在のデータ量（488行、10MB）では完全再計算が十分高速（13秒）
- コストも非常に低い（約0.05円/回）
- パーティショニング・クラスタリングにより、既に効率的

**再検討条件**:
- データ量が100GB以上になった場合
- 実行時間が10分以上かかる場合
- コストが月1000円以上になった場合

### 2026-01: LLM API 選定（Claude 4.5シリーズ）

**背景**: Phase 2でLLMによる投資判断を実装するにあたり、適切なLLM APIを選定する必要があった

**決定**: Anthropic Claude 4.5シリーズを採用
- **本番環境**: Claude Opus 4.5（最高峰の判断品質）
- **PoC/テスト**: Claude Haiku 4.5 → Sonnet 4.5 → Opus 4.5 と段階的に移行

**理由**:
- **判断品質を最優先**: スイングトレードの投資判断という複雑な意思決定タスクでは、判断の質が最重要
- **Extended Thinking**: 多段階推論、複雑な分析に強い（30時間以上の自律的タスクに対応）
- **推論の透明性**: reasoning出力で判断根拠が明確、投資判断の説明責任を果たせる
- **コストは誤差範囲**: 週2回実行で月額$0.21（年間$2.52）、誤判断の損失リスク（数万円〜）と比較すると無視できる
- **段階的な開発**: Haiku（$0.05/月）でPoC、Sonnet（$0.14/月）でプロンプト調整、Opus（$0.21/月）で本番運用
- **モデル切り替えが容易**: 同じAPI（Anthropic）でモデル名を変えるだけ（1行の変更）

**比較検討したAPI**:
- Google Vertex AI（Gemini 2.5 Pro, Gemini 3 Flash）: GCP統合の利便性はあるが、判断品質でClaude Opus 4.5に劣る可能性
- OpenAI（GPT-4o, GPT-5）: 高品質だが、Claude Opus 4.5の方が複雑な推論に強い
- API統合の手間はどのAPIでもほぼ同等（Secret Manager + 環境変数で数行のコード）

**トレードオフ**:
- JSON出力がBeta版（ただし、2025年11月リリースで実績あり、Pydantic対応）
- GCP統合なし（API Key管理が必要だが、Secret Managerで容易に管理可能）
