# タスク一覧の生成

直前の仕様書（または会話内容）を以下のフォーマットでタスク分解してください。

## 出力フォーマット

```markdown
# タスク一覧: [機能名]

| # | タスク | 概要 | 確認方法 |
|---|--------|------|----------|
| 1 | タスク名 | 何をするか | どう確認するか |
| 2 | タスク名 | 何をするか | どう確認するか |
| 3 | タスク名 | 何をするか | どう確認するか |

## 仕様カバレッジ
- [ ] 仕様1: タスク X, Y でカバー
- [ ] 仕様2: タスク Z でカバー
```

## タスク分解の原則

### サイズの目安
- 各タスク: **3時間以内**で完了できる粒度
- 大きすぎる → 分割する
- 小さすぎる → 統合する

### 自然言語で WHAT を記述
- 「何を実現するか」を書く（機能・振る舞い）
- 「どう実装するか」は書かない（ファイル名、関数名は避ける）
- 例: ✅「株価データを BigQuery に保存する」 ❌「bigquery_client.py の insert メソッドを実装」

### 全仕様をカバー
- `/spec` で定義した全ての機能仕様がタスクに紐づいていること
- 「仕様カバレッジ」セクションで漏れがないか確認

### 孤立しないタスク
- 各タスクは前のタスクの成果物を使う
- 最後に統合タスクを入れ、全体が繋がることを確認

## 確認方法の例

- `curl でエンドポイント確認`
- `terraform plan で差分なし`
- `ログ出力を確認`
- `BigQuery でデータ確認`

## 出力後の案内

タスク一覧を出力したら、最後に以下を案内してください：

「タスク1から実装しますか？または並列可のタスクを先に進めますか？」
