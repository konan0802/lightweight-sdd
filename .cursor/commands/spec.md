# 仕様書の生成

ここまでの会話内容を以下のフォーマットで仕様書として整理してください。

## 出力フォーマット

```markdown
# 仕様書: [機能名]

## 概要
何を作るか（1-2文）

## 背景・目的
- なぜ必要か
- 解決したい課題

## 機能仕様

### [機能1の名前]
- **入力**: 何を受け取るか（データ形式、型）
- **出力**: 何を返すか（データ形式、型）
- **正常系**: 期待される動作
- **異常系**: エラー時の振る舞い（リトライ、スキップ、通知など）

### [機能2の名前]
- **入力**: ...
- **出力**: ...
- **正常系**: ...
- **異常系**: ...

## 非機能要件
- 性能、セキュリティ、制約など（該当あれば）

## 技術スタック
- 使用技術とその理由（簡潔に）

## 影響範囲
- 変更が必要なファイル/モジュール
- 既存機能への影響（あれば）

## 完了条件
- [ ] 〇〇が動作すること
- [ ] 〇〇が確認できること

## スコープ外
- 今回やらないこと（明示）
```

## 生成時の原則

### 入出力を明確にする
- 入力: 何を受け取るか（API レスポンス、ユーザー入力、ファイル等）
- 出力: 何を返すか、どこに保存するか
- データ形式: JSON、CSV、BigQuery テーブル等

### 正常系と異常系を分ける
- 正常系: 期待通りに動作した場合の振る舞い
- 異常系: エラー、タイムアウト、データ不正時の振る舞い
- 例: 「API エラー時は 3 回リトライ、失敗したらログ出力して継続」

### 境界条件を考慮する
- データがない場合
- データが大量にある場合
- 重複データがある場合

### コンテキストを活用
- `STEERING.md` を参照し、既存の技術スタックや制約と整合性を取る
- 既存コードへの影響がある場合は「影響範囲」に明記

## 出力後の案内

仕様書を出力したら、最後に以下を案内してください：

「この内容でよければ `/tasks` でタスク分解します」
