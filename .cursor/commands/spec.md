# 仕様書の生成とファイル作成

ここまでの会話内容を仕様書として整理し、`specs/` 配下にファイルを作成します。

## ステップ 1: 仕様書の内容を整理

以下のフォーマットで仕様書を作成してください：

```markdown
# [機能名]

| 項目 | 値 |
|------|---|
| **Status** | draft |
| **作成日** | YYYY-MM-DD |
| **関連チャット** | [タイトル](chat-uuid) |

---

## Spec

### 概要
何を作るか（1-2文）

### 背景・目的
- なぜ必要か
- 解決したい課題

### 機能仕様

#### [機能1の名前]
- **入力**: 何を受け取るか（データ形式、型）
- **出力**: 何を返すか（データ形式、型）
- **正常系**: 期待される動作
- **異常系**: エラー時の振る舞い（リトライ、スキップ、通知など）

### 完了条件
- [ ] 〇〇が動作すること
- [ ] 〇〇が確認できること

### スコープ外
- 今回やらないこと（明示）

---

## Tasks

<!-- /tasks コマンドで生成・追記 -->

| # | タスク | 概要 | 確認方法 | 状態 |
|---|--------|------|----------|------|

---

## Implementation Notes

<!-- 実装中に判明した決定事項・メモを記録 -->

---

## Chat History

| 日付 | 内容 | チャット |
|------|------|---------|
| YYYY-MM-DD | 初回設計 | [タイトル](chat-uuid) |
```

## ステップ 2: ディレクトリ名を提案

以下のフォーマットでディレクトリ名を提案してください：

```
<YYYY-MM>-<slug>
```

- `YYYY-MM`: 今日の日付の年月（例: `2026-03`）
- `slug`: 機能名を英語のケバブケースに変換（例: `user-authentication`）

例: `2026-03-user-authentication`, `2026-03-payment-integration`

既存の `specs/` を確認し、同じ年月のディレクトリがあれば一覧を示した上で提案してください。

## ステップ 3: ファイルを作成

ユーザーがディレクトリ名を確認したら、以下のパスにファイルを作成してください：

```
specs/<YYYY-MM>-<slug>/spec.md
```

## 生成時の原則

### コンテキストを活用
- `STEERING.md` を参照し、既存の技術スタックや制約と整合性を取る
- 既存の `specs/` ディレクトリを確認し、関連するspecがあれば参照

### 関連チャットの記録
- Chat History セクションには現在の会話のコンテキストを記録する
- チャットIDは会話の記録として残すことを促す

### 入出力を明確にする
- 入力: 何を受け取るか（API レスポンス、ユーザー入力、ファイル等）
- 出力: 何を返すか、どこに保存するか

### 正常系と異常系を分ける
- 正常系: 期待通りに動作した場合の振る舞い
- 異常系: エラー、タイムアウト、データ不正時の振る舞い

## 出力後の案内

ファイルを作成したら、最後に以下を案内してください：

「`specs/<YYYY-MM>-<slug>/spec.md` を作成しました。この内容でよければ `/tasks` でタスク分解します」
