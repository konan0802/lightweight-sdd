# タスク実装ガイド

タスクを実装し、完了後に `specs/<YYYY-MM>-<slug>/spec.md` を更新します。

---

## 実装前のチェックリスト

実装を始める前に、以下を確認してください：

### 1. spec.md の確認
- [ ] `specs/<YYYY-MM>-<slug>/spec.md` を開き、対象タスクを確認している
- [ ] タスクの目的（何を実現するか）を理解している
- [ ] 確認方法が明確である

### 2. 依存関係の確認
- [ ] 前提となるタスクが完了している（Tasks テーブルで確認）
- [ ] 必要なリソース（API、データベースなど）が利用可能である

### 3. 技術スタックの確認
- [ ] `STEERING.md` で使用技術・ライブラリを確認している
- [ ] 既存のコードベースの構造を理解している

---

## 実装中のガイドライン

### コーディング規約
- **簡潔さ**: 必要な変更のみを行う
- **可読性**: コメントで意図を説明する
- **保守性**: ベストプラクティスに従う

### 実装の進め方
1. 小さく始める（最小構成で動作確認）
2. テストを書く（可能な場合）
3. エラーハンドリングを追加
4. ログ出力を追加（デバッグ用）

---

## 実装後: spec.md の更新

タスク完了後、**必ず** `specs/<YYYY-MM>-<slug>/spec.md` を以下の通り更新してください：

### 1. Tasks テーブルの状態を更新
```
⬜ 未着手 → ✅ 完了
```

### 2. Implementation Notes に追記
```markdown
### YYYY-MM-DD - タスクX: [タスク名]

- 実装した内容の概要
- 判断・決定したこと（なぜその実装方法を選んだか）
- 気になった点・今後の改善候補

**関連チャット**: [タイトル](chat-uuid)
```

### 3. 全タスク完了時: Status を更新
```
in-progress → done
```

---

## ドキュメント更新の確認

| ドキュメント | 更新が必要なケース |
|--------------|-------------------|
| **STEERING.md** | 技術スタックの追加・変更、アーキテクチャの変更、重要なトレードオフの決定 |
| **README.md** | セットアップ手順の変更、使い方の追加・変更 |
